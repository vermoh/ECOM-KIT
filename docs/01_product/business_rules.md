# Business Rules

## Общие правила платформы
1. Каждый пользователь работает в рамках конкретной организации.
2. Все операции в сервисах tenant-scoped.
3. Если у пользователя нет membership в организации, доступ невозможен.
4. Если срок доступа истёк, доступ блокируется, даже если роль сохранена.
5. Если сервис отключён для пользователя, вход в него невозможен.
6. Deny имеет приоритет над allow.

## Правила AI provider
1. AI provider key может быть задан на уровне организации.
2. AI provider key может быть задан на уровне пользователя только если это разрешено политикой.
3. При наличии user-level key он может иметь приоритет над organization-level key только при явной конфигурации.
4. Ключи не должны быть доступны через frontend или API в открытом виде.
5. Все операции изменения ключей должны логироваться.

## Правила CSV проекта
1. Проект принадлежит одной организации.
2. Один проект может иметь несколько upload attempts, но только один active upload.
3. Enrichment запрещён без approved schema.
4. Export запрещён, если есть unresolved blocking collisions.
5. Все статусы проекта и job должны быть восстановимыми после сбоя.

## Правила schema template
1. SchemaTemplate создаётся на основе category/subcategory и названий SKU.
2. Каждый SchemaField должен иметь тип.
3. Для числовых характеристик должны быть указаны units, если они применимы.
4. Filterable fields должны быть явно помечены.
5. После approval существенные изменения схемы требуют новой версии или повторного подтверждения.

## Правила collision handling
1. Collision создаётся, если confidence ниже порога, данные противоречат схеме или найдено несколько валидных трактовок.
2. Collision может быть blocking или non-blocking.
3. Blocking collision требует ручного решения до экспорта.
4. Все manual overrides должны логироваться.

## Правила доступа и срока действия
1. Срок действия доступа проверяется на backend при каждой критической операции.
2. Истечение доступа не должно давать пользователю возможность создавать новые задания.
3. Поведение активных заданий при истечении доступа задаётся отдельной revocation policy.

## Правила аудита
1. Все операции create/update/delete/approve/run/resolve/export логируются.
2. Лог должен содержать actor, tenant, action, target_type, target_id, timestamp, metadata.
